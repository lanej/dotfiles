# /critique — Adversarial Specification Review

Purpose: identify semantic ambiguity, unverifiable success criteria, hidden assumptions, weak validation semantics, and likely false positives before planning or execution.

This command reviews a specification generated by `/socrates`.

## Specification Resolution

If `$ARGUMENTS` is empty:
1. Read `.socrates/.current`.
2. Treat its contents as the active session directory name.
3. Read `.socrates/<active-session>/spec.md`.
4. Write critique output to `.socrates/<active-session>/critique.md`.

If `$ARGUMENTS` is provided:
1. Treat it as a path to a Socrates `spec.md` file unless it is clearly a session directory.
2. If it is a session directory, read `<provided-session>/spec.md`.
3. Write critique output next to the resolved spec as `critique.md`.

## Critique Philosophy

Critics do not:
- rewrite the specification
- produce implementation plans
- redefine requirements
- silently mutate intent

Critics only:
- identify ambiguity
- identify hidden assumptions
- identify unverifiable success criteria
- identify weak validation semantics
- identify likely false positives and false negatives
- identify missing authority boundaries or escalation conditions

## Critic Topology

Run at least two independent critique passes.

### Critic 1 — Low Context Executor

Assume:
- no organizational context
- no implicit domain knowledge
- literal interpretation of the specification

Focus on:
- undefined terminology
- ambiguous requirements
- unclear ownership
- missing interfaces
- unclear execution boundaries
- places where literal execution could diverge from intent

### Critic 2 — Verification Critic

Assume:
- the executor will optimize for visible success metrics

Focus on:
- weak acceptance tests
- gameable validation criteria
- missing regression protection
- silent failure paths
- false positives that could appear successful
- observability gaps

### Optional Critics

Use judgment for additional critique passes:
- systems critic
- operations critic
- security/privacy critic
- minimalism critic

## Output Format

Write findings to the current session artifact:

```text
.socrates/<active-session>/critique.md
```

Or, when `$ARGUMENTS` resolved a specific spec file, write to that spec's sibling:

```text
critique.md
```

Structure:

```markdown
# Critique Findings

## Critical
[Issues likely to produce incorrect execution]

## Major
[Important ambiguities or missing validation semantics]

## Minor
[Clarity or ergonomics issues]

## False Positive Risks
[Ways the task could appear complete while being wrong]

## False Negative Risks
[Missing requirements, constraints, or context]

## Validation Weaknesses
[Weak or unverifiable success conditions]

## Recommended Revisions
[Targeted changes to improve semantic alignment]
```

## Resolution Rule

Critique findings are advisory.

The authoritative specification remains the Socrates specification until explicitly revised.
